┌─────────────────────────────────────────────────────────────────────┐

## Q2: Describe a time you had to influence without authority

### VP-Level Answer:

```
┌─────────────────────────────────────────────────────────────────────┐
│               INFLUENCE: GETTING MODEL RISK BUY-IN                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   SITUATION                                                          │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │ We needed Model Risk Management (MRM) to approve our ML      │   │
│   │ pricing model. MRM was skeptical of ML for pricing -         │   │
│   │ they'd only approved rule-based systems before. They had     │   │
│   │ veto power; I had no authority over them.                    │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│   TASK                                                               │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │ Get MRM approval for the first ML-based pricing model        │   │
│   │ at the bank, without any formal authority over the           │   │
│   │ approval process.                                            │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│   MY INFLUENCE APPROACH                                              │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │                                                              │   │
│   │   1. UNDERSTAND THEIR PERSPECTIVE                            │   │
│   │   ┌─────────────────────────────────────────────────────┐   │   │
│   │   │ • Met with MRM lead 1:1 before formal review          │   │   │
│   │   │ • Asked: "What would make you comfortable with ML?"   │   │   │
│   │   │ • Learned their concerns: explainability, stability,  │   │   │
│   │   │   fairness, and audit trail                           │   │   │
│   │   └─────────────────────────────────────────────────────┘   │   │
│   │                                                              │   │
│   │   2. ADDRESS CONCERNS PROACTIVELY                            │   │
│   │   ┌─────────────────────────────────────────────────────┐   │   │
│   │   │ • Built explainability (SHAP) into the system        │   │   │
│   │   │ • Added stability monitoring with automatic alerts    │   │   │
│   │   │ • Implemented fairness testing before they asked      │   │   │
│   │   │ • Created detailed audit logging                      │   │   │
│   │   └─────────────────────────────────────────────────────┘   │   │
│   │                                                              │   │
│   │   3. MAKE THEM PARTNERS, NOT GATEKEEPERS                     │   │
│   │   ┌─────────────────────────────────────────────────────┐   │   │
│   │   │ • Invited MRM to architecture review meetings         │   │   │
│   │   │ • Asked for their input on monitoring thresholds      │   │   │
│   │   │ • Gave them credit for governance framework design    │   │   │
│   │   └─────────────────────────────────────────────────────┘   │   │
│   │                                                              │   │
│   │   4. BUILD CREDIBILITY INCREMENTALLY                         │   │
│   │   ┌─────────────────────────────────────────────────────┐   │   │
│   │   │ • Started with shadow mode (model scores but          │   │   │
│   │   │   doesn't make decisions)                             │   │   │
│   │   │ • Showed 3 months of stable performance               │   │   │
│   │   │ • Proposed gradual rollout with clear exit criteria   │   │   │
│   │   └─────────────────────────────────────────────────────┘   │   │
│   │                                                              │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│   RESULT                                                             │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │ • Model approved - first ML pricing model at the bank        │   │
│   │ • MRM lead became an advocate for ML in other areas          │   │
│   │ • Governance framework became template for future models     │   │
│   │ • Approval process went from 6 months to 3 months for        │   │
│   │   subsequent models                                          │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│   KEY INSIGHT                                                        │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │ "Influence comes from understanding what the other party     │   │
│   │  needs to say yes. MRM needed to feel confident they         │   │
│   │  weren't taking on excessive risk. By addressing their       │   │
│   │  concerns before they raised them, I made it easy for        │   │
│   │  them to approve."                                           │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

## Q4: How do you handle conflicting priorities from stakeholders?

### VP-Level Answer:

```
┌─────────────────────────────────────────────────────────────────────┐
│             CONFLICTING PRIORITIES: FRAMEWORK                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   MY APPROACH                                                        │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │                                                              │   │
│   │   Step 1: MAKE TRADE-OFFS EXPLICIT                           │   │
│   │   ┌─────────────────────────────────────────────────────┐   │   │
│   │   │ "I create visibility into the trade-off. Instead of  │   │   │
│   │   │  quietly choosing, I show stakeholders: if we do A,  │   │   │
│   │   │  we can't do B in the same timeframe."               │   │   │
│   │   └─────────────────────────────────────────────────────┘   │   │
│   │                                                              │   │
│   │   Step 2: USE A PRIORITIZATION FRAMEWORK                     │   │
│   │   ┌─────────────────────────────────────────────────────┐   │   │
│   │   │                                                       │   │   │
│   │   │   Impact   │ High │ High │ Low  │ Low  │              │   │   │
│   │   │   Urgency  │ High │ Low  │ High │ Low  │              │   │   │
│   │   │   ─────────│──────│──────│──────│──────│              │   │   │
│   │   │   Priority │  1   │  2   │  3   │  4   │              │   │   │
│   │   │                                                       │   │   │
│   │   │   But also consider: strategic alignment,             │   │   │
│   │   │   dependencies, risk, stakeholder relationship        │   │   │
│   │   │                                                       │   │   │
│   │   └─────────────────────────────────────────────────────┘   │   │
│   │                                                              │   │
│   │   Step 3: ESCALATE WHEN NEEDED                               │   │
│   │   ┌─────────────────────────────────────────────────────┐   │   │
│   │   │ "If two VP-level stakeholders both insist their      │   │   │
│   │   │  priority comes first, I don't play politics.        │   │   │
│   │   │  I escalate to their shared manager with clear       │   │   │
│   │   │  options and trade-offs."                            │   │   │
│   │   └─────────────────────────────────────────────────────┘   │   │
│   │                                                              │   │
│   │   Step 4: COMMUNICATE DECISIONS CLEARLY                      │   │
│   │   ┌─────────────────────────────────────────────────────┐   │   │
│   │   │ "Once priorities are set, I communicate to all       │   │   │
│   │   │  stakeholders what we're doing and why. No one       │   │   │
│   │   │  should be surprised."                               │   │   │
│   │   └─────────────────────────────────────────────────────┘   │   │
│   │                                                              │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│   CONCRETE EXAMPLE                                                   │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │ Situation: Three business units wanted different models      │   │
│   │ deployed in Q1, but we only had capacity for two.            │   │
│   │                                                              │   │
│   │ What I did:                                                  │   │
│   │ 1. Quantified impact of each ($ opportunity, risk reduction) │   │
│   │ 2. Created a one-page comparison showing trade-offs          │   │

│                                                                      │
│   KEY INSIGHT                                                        │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │ "My job isn't to make everyone happy - it's to make trade-  │   │
│   │  offs visible and ensure decisions are made consciously.    │   │
│   │  Stakeholders respect that even when they don't get what    │   │
│   │  they want."                                                │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

## Q5: Tell me about a time you had to make a decision with incomplete information

### VP-Level Answer:

```
┌─────────────────────────────────────────────────────────────────────┐
│          DECISION: TECHNOLOGY BET ON TRANSFORMERS                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   SITUATION                                                          │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │ In 2020, we needed to choose architecture for our pricing   │   │
│   │ model. Options were: traditional ML (XGBoost), LSTM, or     │   │
│   │ the then-newer transformer architecture.                     │   │
│   │                                                              │   │
│   │ Incomplete information:                                      │   │
│   │ • Limited production examples of transformers on tabular    │   │
│   │ • Uncertain regulatory acceptance of complex models         │   │
│   │ • Team had no transformer experience                        │   │
│   │ • ROI was theoretical, not proven                           │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│   MY DECISION FRAMEWORK                                              │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │                                                              │   │
│   │   1. IDENTIFY WHAT I DON'T KNOW                              │   │
│   │   ┌─────────────────────────────────────────────────────┐   │   │
│   │   │ • Will transformers work on our tabular data?        │   │   │
│   │   │ • Can we get regulatory approval?                    │   │   │
│   │   │ • Can we build the capability in time?               │   │   │
│   │   └─────────────────────────────────────────────────────┘   │   │
│   │                                                              │   │
│   │   2. ASSESS WHAT I CAN LEARN QUICKLY                         │   │
│   │   ┌─────────────────────────────────────────────────────┐   │   │
│   │   │ • Technical feasibility: 2-week proof of concept     │   │   │
│   │   │ • Regulatory path: Conversation with model risk      │   │   │
│   │   │ • Team capability: Could hire or train               │   │   │
│   │   └─────────────────────────────────────────────────────┘   │   │
│   │                                                              │   │
│   │   3. EVALUATE REVERSIBILITY                                  │   │
│   │   ┌─────────────────────────────────────────────────────┐   │   │
│   │   │ If transformers fail, can we fall back to XGBoost?   │   │   │
│   │   │ Yes - the feature engineering work applies to both.  │   │   │
│   │   │                                                       │   │   │
│   │   │ Cost of being wrong: 2-3 months of R&D                │   │   │
│   │   │ Cost of being right but late: Competitor advantage    │   │   │
│   │   └─────────────────────────────────────────────────────┘   │   │
│   │                                                              │   │
│   │   4. MAKE A BOUNDED BET                                      │   │
│   │   ┌─────────────────────────────────────────────────────┐   │   │
│   │   │ Decision: Pursue transformer, but with exit criteria │   │   │
│   │   │                                                       │   │   │
│   │   │ • 6-week checkpoint: >5% improvement over XGBoost?    │   │   │
│   │   │ • If no: Switch to XGBoost                            │   │   │
│   │   │ • If yes: Continue investment                         │   │   │
│   │   └─────────────────────────────────────────────────────┘   │   │
│   │                                                              │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│   RESULT                                                             │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │ • 6-week checkpoint: 12% improvement - continued            │   │
│   │ • Final model: 40% improvement in margin capture            │   │
│   │ • Team built transformer expertise                          │   │
│   │ • Became template for other ML initiatives                  │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│   KEY INSIGHT                                                        │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │ "The goal isn't to be certain - it's to structure the       │   │
│   │  decision so we learn fast and can course-correct.          │   │
│   │  Bounded bets with clear checkpoints turn uncertain         │   │
│   │  decisions into experiments."                               │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

2. **Judgment**: How you navigate complex situations
3. **Influence**: Getting things done without formal authority
4. **Communication**: Delivering difficult messages effectively
5. **Adaptability**: Changing course when needed

Key patterns for VP-level answers:
- Use specific examples with concrete outcomes
- Show the reasoning behind your actions
- Demonstrate learning and growth
- Take appropriate responsibility
- Connect experiences to transferable principles
